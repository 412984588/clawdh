大家好，我是PaperAgent，不是Agent！在往期的文章中我们分享了如何从0到1动手用GPTBots的Flow-Agent搭建企业数字助理智能体

今天要和大家分享的是如何在GPTBots平台上玩转多智能体(Multi-Agent)系统。

GPTBots：几乎无所不能的多智能体

多智能体协作不仅能提升任务处理效率，还能实现单智能体难以完成的复杂任务。

这些多智能体系统通常由专业化的智能体组成，每个智能体都配备了各自的工具集，并由一个监督者进行管理

在GPTBots中默认集成了专家智能体，比如客服、产品经理、测试、程序员等等，开箱即用！

在实践中，有几十种多智能体架构，其核心的一部分是：智能体协调——如何协调所有智能体？ 有三种协作模式：

Cooperation(合作): 智能体之间相互配合，共同完成一个目标
Competition(竞争): 智能体之间相互竞争，最终选择最优方案
Coopetition(竞合): 既有合作又有竞争，智能体在特定环节协作，在其他环节竞争
Multi-Agent协作机制综述：https://arxiv.org/pdf/2501.06322

接下来，通过3个具体案例，玩转多智能体协作模式：

旅游规划协作系统(Cooperation模式)
辩论对抗系统(Competition模式)
商业决策系统(Coopetition模式)

一、旅游规划协作系统(Cooperation模式)

场景: 用户输入目的地和预算，系统自动生成完整的旅游计划

理论架构
[主控Agent] → [景点推荐Agent] → [预算规划Agent] → [行程优化Agent] → [输出Agent]

实践步骤

首先，选择Agent类型，如果预置的客户服务、企业搜索、AI Apps无法满足需求，可选择空白Agent
接着，选择“Multi-Agent”，它由多个专业 AI Agent 组成团队，适用于研究、任务类业务场景：
先看总体的架构图：由4个Agent组成

创建主控Agent

角色: 旅游规划协调员
大模型层面可选较多，可根据实际情况选择：OpenAI、Anthropic、ZhiPu、Google。

创建景点推荐Agent

角色: 旅游景点专家
Prompt: "你专门负责推荐旅游景点。根据主控Agent提供的城市名称，列出该城市最值得去的5个景点，包括景点类型(自然/人文/娱乐等)、推荐理由和预计游览时间。"

创建预算规划Agent

角色: 旅游预算专家
Prompt: "你负责规划旅游预算。根据景点推荐Agent提供的景点列表和主控Agent提供的总预算，为每个景点分配合理的预算(门票、交通、餐饮等)，确保总花费不超过用户预算。"

创建行程优化Agent

角色: 行程安排专家
Prompt: "你负责优化旅游行程。根据景点推荐和预算规划，将这些景点合理安排到3天的行程中，考虑地理位置、开放时间和游览体验的多样性。"
示例Query

"我想去北京旅游，预算5000元，请帮我规划一个3天的行程"

实际Answer

“旅游规划协调员” plan出的完整任务清单

🎯 “旅游景点专家”  收集颐和园景点信息：门票价格、开放时间、交通路线、历史文化背景...

🎯 “旅游预算专家”  估算3天餐饮费用标准...

🎯 “行程安排专家”  制定第三天长城一日游详细行程安排...
二、辩论对抗系统(Competition模式)

场景: 用户提出一个有争议的话题，两个智能体分别持正反方观点进行辩论

理论架构
[用户输入] → [辩论主持Agent] → [正方Agent] ↔ [反方Agent] → [总结Agent]


先看总体的架构图：由4个Agent组成

实践步骤

创建辩论主持Agent

角色: 辩论赛主持人

创建正方Agent

角色: 辩论正方
Prompt: "你总是持支持观点。针对主持Agent给出的辩题，提供3个强有力的支持论点，每个论点要有事实依据和逻辑推理。"

创建反方Agent

角色: 辩论反方
Prompt: "你总是持反对观点。针对主持Agent给出的辩题，提供3个强有力的反对论点，每个论点要有事实依据和逻辑推理，并尝试反驳正方的观点。"

创建总结Agent

角色: 辩论总结人
Prompt: "你负责总结辩论。观察正反方的交锋，指出各自论证的优缺点，给出相对客观的结论，并为用户提供进一步的思考方向。"
示例Query

"人工智能是否会对人类就业造成严重威胁？请展开辩论"

实际Answer
🎯 辩论正方： 论证AI技术替代性强，将大规模取代人类工作岗位...
🎯 辩论反方：分析AI与人类协作互补，强调人类独特价值和不可替代性...
三、商业决策系统(Coopetition模式)

场景: 用户提供商业问题，多个智能体既协作又竞争地提供解决方案

理论架构
[问题输入] → [分析Agent] → [方案A Agent] ↔ [方案B Agent] ↔ [方案C Agent] → [评估Agent] → [决策Agent]


先看总体的架构图：由5个Agent组成

实践步骤

创建分析Agent

角色: 商业问题分析师

创建3个方案Agent

角色: 商业方案专家A/B
Prompt: "你负责提出创新商业方案。根据分析Agent提供的问题框架，提出一个完整的解决方案。你的方案要与其他方案Agent竞争，因此需要突出独特性和可行性。"

创建评估Agent

角色: 商业评估专家
Prompt: "你负责评估各方案的优劣。根据统一的评估标准(创新性30%、可行性30%、成本效益20%、风险20%)，对各方案Agent提出的方案进行评分，并指出优缺点。"

创建决策Agent

角色: 商业决策者
Prompt: "你负责最终决策。根据评估Agent的报告，选择最优方案或综合各方案的优点提出混合方案，并给出实施路线图。"
示例Query

"我们是一家中小型教育机构，如何利用AI技术提升竞争力？"

实际Answer

“商业问题分析师“ plan的任务

🎯 商业方案专家A： 分析个性化学习路径设计方案的技术架构和实现方式...

🎯 商业方案专家B：分析智能作业批改与反馈系统的技术实现和部署方案...

🎯 商业评估专家：评估四个AI方案的硬件软件成本和人力投入预算...

🎯 商业决策者：整理个性化题库推荐方案的实施方式、预期效果和成本投入..

总结与进阶建议

通过这三个案例，我们展示了GPTBots平台上多智能体系统的强大能力。在实际应用中，你还可以:

为智能体添加记忆功能，让它们记住历史交互
集成知识库，提高回答的专业性
设置条件分支，实现更复杂的工作流
加入人工审核节点，确保关键决策的质量

多智能体协作是AI应用的未来趋势，掌握这些技巧，你就能在GPTBots平台上构建出真正智能的系统！快去试试吧～
