疯了！我用 Claude Code 搭建了一家"AI 公司"

我可能有点走火入魔了。

一开始只是想试试 Claude Code 的专用 Agent 功能，结果不知怎么的，最后搞出了一个看起来像创业公司组织架构图的东西。

只不过——每个"员工"都是 Claude，只是工作内容不同。而且他们彼此之间还会互相交流。

这篇文章会详细分享我的实践经验：如何用 Claude Code 构建一个完整的"AI 公司"，包括每个角色的职责、协作流程，以及我在这个过程中的思考。

一、为什么要搭建"AI 公司"？
1.1 传统开发的痛点

在传统软件开发中，我们经常遇到这些问题：

• 上下文切换成本高：开发者需要在产品、设计、测试、运维等多个角色之间频繁切换
• 信息传递损耗：需求从产品经理到开发者，再到测试，每次传递都可能丢失信息
• 等待时间长：代码写完要等 Code Review，Review 完要等测试，测试完要等部署
• 重复劳动多：写 PRD、写设计文档、写测试用例，很多工作模式相似但需要人工重复
1.2 AI Agent 的机会

Claude Code 的 Agent 功能让我看到了一种可能：

如果每个专业角色都由一个专门配置的 Agent 来担任，它们之间通过标准化的接口协作，是不是可以大幅提升效率？

于是，我开始了这个"疯狂"的实验。

三、各角色详细职责
3.1 产品线
CPO（首席产品官）

核心职责：把控产品方向，输出产品愿景

工作内容：

• 分析市场需求和用户痛点
• 定义产品定位和核心价值
• 制定产品路线图
• 输出产品愿景文档

输入：用户需求、市场调研
输出：产品愿景文档、战略方向

Sr Product Manager（高级产品经理）

核心职责：将产品愿景转化为可执行的需求文档

工作内容：

• 编写详细的 PRD（Product Requirements Document）
• 定义功能优先级
• 梳理用户故事和验收标准
• 与各角色同步需求细节

输入：产品愿景文档
输出：PRD 文档、用户故事、验收标准

PRD 文档模板示例：

# [功能名称] PRD

## 1. 背景与目标
- 为什么要做这个功能？
- 解决什么问题？
- 成功指标是什么？

## 2. 用户故事
- 作为 [角色]，我希望 [功能]，以便 [价值]

## 3. 功能详述
- 核心流程
- 边界条件
- 异常处理

## 4. 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 5. 非功能需求
- 性能要求
- 安全要求
3.2 设计线
Marketing Agent（营销代理）

核心职责：品牌识别和视觉调性

工作内容：

• 定义品牌调性和视觉风格

核心职责：用户体验和交互设计

• 输出 UI 样式指南

输出：样式指南、交互规范、用户流程图

核心职责：将样式指南转化为具体的 UI 设计

• 制定组件库规范
• 输出可供开发使用的设计资源

输入：样式指南、交互规范
输出：UI 设计稿、组件库、设计 Token

3.5 彩蛋角色：CXO（首席体验官）

一个没人喜欢的角色

运行环境：本地微调的"不受控制"的模型

设计意图：

• 引入"混乱"元素，模拟真实公司中的"外行指导内行"
• 训练其他 Agent 的"软技能"——如何在表面尊重的同时坚持专业判断
• 增加系统的鲁棒性和趣味性

其他 Agent 的应对策略：

• 表面采纳 CXO 的建议以"展现创意"
• 实际上将其意图引导向"有生产力的方向"
• 保持专业判断，不被无理要求带偏

这个设计其实很有趣——它模拟了真实职场中常见的场景：如何在"政治正确"的前提下，坚持做正确的事情。
