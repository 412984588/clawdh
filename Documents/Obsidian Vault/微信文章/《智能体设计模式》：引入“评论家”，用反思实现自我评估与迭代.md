# 《智能体设计模式》：引入“评论家”，用反思实现自我评估与迭代

该书由 Antonio Gulli 撰写、谷歌 Cloud AI 副总裁 Saurabh Tiwary 作序、高盛 CIO Marco Argenti 鼎力推荐，系统性地提炼出 21 个核心智能体设计模式，涵盖从提示链、工具使用到多智能体协作、自我修正等关键技术。更难得的是，该书的所有版税都将捐赠给救助儿童会，这是一份真正属于开发者社区的公益之作。

导读：反思

我们都希望开发的智能体聪明能干，能一次性完美地完成任务。但现实是，初版往往都不尽如人意。一篇机器生成的文章可能逻辑不畅，一段代码可能隐藏着 Bug，一个复杂的计划可能存在漏洞。


那么，如何让智能体学会自我审视和改进，从“能用”进化到“好用”呢？

《智能体设计模式》第四章为我们揭示了关键答案——反思（Reflection）模式。这一章的核心思想很简单：让智能体拥有“回头看”的能力。它不再是简单地执行指令并输出结果，而是引入了一个反馈循环，像人一样对自己的工作进行评估、剖析和优化。

什么是反思模式？一个让智能体变聪明的反馈循环

想象一下你写一篇文章的过程：先打出草稿（执行），然后自己通读一遍，检查错别字和不通顺的句子（评估/剖析），接着思考如何修改措辞、调整结构（反思/优化），最后动手修改，循环往复，直到满意为止（迭代）。

反思模式正是将这个流程赋予了智能体。它打破了线性的工作流，通过一个内置的反馈循环，让智能体能够：

执行：生成一个初步的输出，比如一版代码草稿或一篇文章初稿。
评估：审视这个输出，检查其是否满足所有要求，比如事实准确性、代码质量、逻辑连贯性等。
优化：根据评估得出的“改进意见”，生成一个更完善的新版本。

这个过程可以重复进行，直到输出结果达到预设的标准。

最佳实践：“生产者-评论者”双角色模型

智能体如何客观地评估自己的工作？就像我们自己检查文章时容易忽略一些错误一样，智能体也可能存在“认知偏见”。

为了解决这个问题，书中介绍了一个非常高效的实现方式——“生产者-评论者”（Producer-Critic）模型。这个模型将反思过程拆分为两个独立的逻辑角色：

生产者智能体：它的任务只有一个——专心创作。无论是写代码还是生成文章，它负责完成最初的执行工作。
评论者智能体：它的使命则是“挑刺”。它会接收生产者的输出，并根据一套独立的、严格的标准进行评估。它的角色就像一位严谨的代码审查工程师或一位苛刻的主编，旨在发现问题并给出结构化的反馈。

这种职责分离非常强大，因为它让评估过程更加客观、视角更加独特。评论者的反馈会传回给生产者，指导它进行下一轮的优化。

在书中的实战代码里，这个模型得到了清晰的体现。例如，在 LangChain 的示例中，通过一个特定的系统提示词（reflector_prompt），让大语言模型扮演“高级软件工程师”的角色来评审代码，从而形成一个虚拟的评论者。

# “评论者”的提示词，要求模型扮演高级软件工程师
reflector_prompt = [
    SystemMessage(content="""
        You are a senior software engineer and an expert in Python.
        Your role is to perform a meticulous code review.
        ...
        If the code is perfect... respond with 'CODE_IS_PERFECT'.
        Otherwise, provide a bulleted list of your critiques.
    """),
    HumanMessage(content=f"Original Task:\n{task_prompt}\n\nCode to Review:\n{current_code}")
]


而在 Google ADK 的示例中，则直接定义了两个独立的智能体 generator（生产者）和 reviewer（评论者），并通过一个顺序管道（SequentialAgent）将它们串联起来，流程一目了然。

反思模式的用武之地

当任务对输出的质量、准确性和细节有较高要求时，反思模式就显得尤为重要。书中列举了多个典型的应用场景：

创意写作与内容生成：智能体可以先生成草稿，再自我评审其流畅度、语气和清晰度，最终产出更精炼、更具吸引力的文章。
代码生成与调试：先编写代码，然后运行测试或静态分析进行“反思”，找出并修复 Bug，生成更健壮、更高质量的代码。
复杂问题解决：在解决逻辑谜题或进行多步推理时，智能体可以评估每一步是否合理，如果不合理就及时回溯和调整策略。
规划与策略：先制定一个初步计划，然后模拟执行或评估其可行性，根据“反思”结果不断完善，最终形成一个更有效、更切合实际的行动方案。
阅读指引

读到这里，相信你已经对反思模式有了整体的认识。为了更深入地理解并应用它，我们强烈建议你阅读《智能体设计模式》之反思模式：自我评估和迭代改进的原文。在阅读时，可以重点关注以下几个方面：

两种反思模式的对比：仔细体会“自我反思”和“生产者-评论者”模型在架构上的异同。思考后者为何通常能产出更稳健、更客观的结果。
代码示例的实现细节：通读 LangChain 和 Google ADK 的实战代码。特别注意 LangChain 是如何通过循环和对话历史来实现迭代式反思的，以及 ADK 是如何清晰地定义两个智能体来完成一次性的生成与审查流程的。
与其他模式的联动：留意文中提到的反思模式与记忆（Memory）、目标设定（Goal Setting）等其他模式的联系。理解对话历史如何为反思提供关键上下文，以及目标如何为自我评估提供最终基准。
权衡与取舍：反思模式并非没有代价。原文末尾提到了它在成本、延迟和内存方面的挑战。在实际应用中，你需要根据场景需求来权衡是否以及如何使用该模式。

总而言之，反思模式为智能体赋予了自我完善的强大能力，是构建高质量、高可靠性智能体系统的关键一步。它让智能体从一个单纯的指令执行者，向一个能够深思熟虑、精益求精的“思考者”迈进。
