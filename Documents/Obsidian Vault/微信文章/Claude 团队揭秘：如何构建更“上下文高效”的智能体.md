# Claude 团队揭秘：如何构建更“上下文高效”的智能体

> 作者: PageIndex | 发布日期: 2025年11月14日 02:34
> 原文链接: [点击查看](https://mp.weixin.qq.com/s/7tyL30qbXfr8X72_2z7wSA)

---

当智能体直接调用工具时，每次调用的定义与结果都会占用上下文窗口。如果让智能体自己“写代码”来调用工具，它的扩展性和效率就能显著提升。Anthropic 的最新博客《通过代码执行使用 MCP：让智能体更高效（Code execution with MCP: Building more efficient agents）》介绍了如何利用 MCP 实现这一点。
Model Context Protocol (MCP) 是一种开放标准，用于让 AI 智能体与外部系统互通。过去，连接智能体与工具或数据需要针对每种组合编写定制集成代码，导致重复劳动和碎片化问题，使得系统难以规模化扩展。MCP 提供了统一的连接方式，开发者只需在智能体中实现一次 MCP，就能访问整个生态系统的集成资源。
自 MCP 于 2024 年 11 月推出以来，业界采用速度惊人：社区已经构建了上千个 MCP 服务器，主流编程语言都提供了 SDK，各大公司也逐渐将 MCP 视为连接工具与数据的事实标准。
如今，开发者常常让智能体同时访问数百甚至上千种工具、跨多个 MCP 服务器。然而，随着连接规模扩大，加载所有工具定义并将中间结果通过上下文传递，智能体运行会变慢，成本也显著上升。
下文将展示如何通过代码执行（code execution）让智能体更高效地与 MCP 服务器交互：在处理更多工具的同时，用更少的 token 完成任务。
过多的 token 消耗降低了智能体效率
随着 MCP 的广泛使用，常见的性能瓶颈主要来自两个方面：
工具定义让上下文过载
很多 MCP 客户端会在启动时，将所有工具定义直接加载进上下文，供模型使用。例如像从 Google Drive 获取文档或在 Salesforce 更新记录这样的工具，通常都会包含详细的参数说明和返回格式。
这些定义看似无害，但它们会占据大量上下文空间，导致响应变慢、计算成本上升。如果智能体连接了几千个工具，它可能需要在真正开始执行任务前，先处理几十万个 token。
2. 中间结果占用更多 token
另一种常见情况是：智能体直接在上下文中处理工具的中间结果。
例如，当用户让智能体“从 Google Drive 下载会议记录并附加到 Salesforce 客户记录”时，模型会先调用一个工具获取完整的会议文字，再调用另一个工具把相同内容上传到目标系统。
在这个过程中，会议记录的完整内容要经过模型两次。对于一场两小时的会议，这可能意味着多处理 50,000 个 token。如果是更大的文档，甚至可能超出上下文限制，让任务中断。
此外，在复制大数据结构时，模型更容易出错，比如可能出现字段遗漏或格式错乱等问题。
图示：MCP 客户端如何与 MCP 服务器和大语言模型（LLM）协作。
客户端将工具定义加载进模型上下文，并在每次调用间通过消息循环传递结果。
让智能体通过代码使用 MCP：更高效地管理上下文
如今，许多智能体开始支持代码执行环境。在这种模式下，我们可以把 MCP 服务器当作“代码 API”，而不是“直接工具调用”。这样，智能体能自己写代码与 MCP 服务器交互。
这种方式同时解决了两个问题：一方面，智能体只需加载所需的工具定义；另一方面，它能在执行环境中预处理数据，再返回简化结果给模型。
比如，可以把所有 MCP 工具映射成一个文件树结构。用 TypeScript 来实现大致如下：
servers
├── google-drive
│   ├── getDocument.ts
│   ├── ... (other tools)
│   └── index.ts
├── salesforce
│   ├── updateRecord.ts
│   ├── ... (other tools)
│   └── index.ts
└── ... (other servers)
每个文件代表一个可调用的工具，开发者可以为每个工具编写独立的代码文件，例如一个函数用于从 Google Drive 读取文档，另一个函数用于在 Salesforce 中更新记录。以“从 Google Docs 读取会议记录并上传到 Salesforce 客户信息”为例，智能体只需先调用读取函数获取内容，再调用更新函数将文本写入对应的客户记录。整个过程无需让模型在上下文中逐步传递中间结果，数据会在代码执行环境中自动流转，形成一个高效的自动化链路。
智能体通过浏览文件系统（例如 ./servers/）即可找到可用的工具文件。它只需加载当前任务需要的部分，从而把 token 使用量从 15 万减少到约 2 千，节省约 98.7% 的时间与成本。
Cloudflare 也有类似发现，并称这种方法为“Code Mode”。核心思想一致：既然大语言模型擅长写代码，就让它们写代码来操作 MCP 服务器。
让智能体通过代码使用 MCP 的优势
渐进式加载（Progressive disclosure）
模型非常擅长探索文件系统。把工具定义变成代码文件后，模型可以按需读取，而无需一次性加载全部定义。开发者还可以添加一个 search_tools 功能，让智能体通过关键字查找需要的工具，只加载必要部分，从而进一步节省上下文空间。
2. 上下文更高效的结果处理
当处理大型数据集时，智能体可以在代码执行环境中先对结果进行过滤或转换，再返回给模型。这样，模型只需查看经过筛选后的少量结果，而无需处理整个数据集。同样的思路也适用于数据汇总、跨来源数据整合，或提取特定字段等操作，所有这些都能在不占用过多上下文窗口的情况下完成。
3. 更智能的控制逻辑
智能体可以像普通程序一样使用循环、条件判断和异常处理。相比通过模型逐步判断，这样的循环逻辑更快、更高效，也能减少响应延迟。
4. 数据隐私更安全
代码执行让中间数据默认保存在执行环境中，模型只接收显式输出的数据。在这种情况下，MCP 客户端会自动把个人敏感信息（PII）标记化处理，模型看到的只是模糊化后的字段，而真实数据仍安全地流向 Salesforce。
5. 状态保存与技能积累（Skills）
代码执行配合文件系统访问，让智能体能持久保存状态。它可以将中间结果写入文件，稍后继续执行；也能把常用逻辑保存为可复用的函数。
这就是“技能（Skills）”的概念：可复用的指令、脚本和资源，帮助模型在特定任务上越用越聪明。随着时间推移，智能体将积累越来越多的“技能模块”，成为真正能自主提升能力的系统。
Anthropic 提出的这种模式，把成熟的软件工程范式引入到智能体世界：不再让模型被动地接收所有信息并一次性处理，而是让它通过编写和运行代码，在一个结构化的工具空间中按需调用 MCP 等接口，完成检索、操作与编排。这种方式既节省上下文与算力，也让 AI 的行为更接近真正的软件工程师。不过，这套方法主要服务于高度结构化的任务场景，例如代码执行、文件操作、表单处理、API 编排等，这些场景的输入输出边界清晰、操作语义稳定、逻辑路径相对可预期。
当我们尝试把同样的“结构化入口 + 按需加载 + 渐进式披露”的思路，迁移到非结构化的长文档时，挑战也随之而来：在没有固定接口、没有清晰边界的成千上万字文本里，模型如何才能像人类一样有条理地翻找、推理并整合信息？
PageIndex：面向长文档的结构化上下文管理框架
PageIndex 与上述基于 MCP / Claude Skills 的渐进式上下文管理的理念不谋而合，它是全球首个针对长文档的 In-Context Index （上下文内索引）系统，通过对文档预生成结构化摘要目录树（Table-of-Contents Tree），将文档的 context 转化为一个 LLM 可直接读取的、渐进式的“上下文内索引”。
在推理阶段，LLM 不再直接读取冗长的全文，而是先加载这一目录树结构。这让模型能够像人类一样，通过目录理解并导航文档结构、逐层定位到相关章节，再按需局部展开相关内容。
这种方法的优点体现在以下几点：
高效压缩：树状索引的长度远小于原文档，但保留了文档的结构框架与关键信息点，实现高压缩率的同时保持关键上下文关联，从而显著降低模型的上下文负担；同时，文档的 context 内容可按需进行渐进式、局部展开。
层级化推理：索引中的章节层次构成了清晰的语义层级，为模型提供上下文贯通的“推理路径（reasoning path）”，使其能够在上下文内进行渐进地、逐层深入的定位、归纳与分析；
上下文内检索（In-Context Retrieval）：索引直接存在于模型上下文中，无需外部向量数据库或检索模块，实现检索与推理的一体化处理；
可解释与可溯源：模型的推理路径对应目录节点，可直观追踪推理来源，使检索具备更好的可控性与透明度。
因此，PageIndex 不仅是一种长文档索引与检索技术，更是一种面向长任务的上下文工程框架。它以结构化索引代替无差别的上下文堆叠，以保留关键信息的稀疏表示替代冗余输入，让 LLM 能够在有限窗口与注意力预算下，实现持续而连贯的上下文理解与推理。
