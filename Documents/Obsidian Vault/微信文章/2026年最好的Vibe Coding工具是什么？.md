2025年11月4日，知名创投人Greg Isenberg和Ras Mic在YouTube上做了一个播客，讨论2026年最佳Vibe Coding工具，整个内容干货满满，所以跟大家分享一下。

在分享之前，先做一些基本的解释：

Vibe Coding到底是什么

首先，解释下，Vibe Coding 是一种让人完全沉浸在与AI共同创造的流动状态中的编程方式。在这种方式下，开发者更多的是提供抽象的、高层次的提示，而不是逐行编写代码。其核心思想是将“写代码”这一动作淡化，转而专注于创意的流动和实验性迭代。Vibe Coding 强调快速验证想法、搭建原型以及探索性项目的高效性，特别适用于构建最小可行产品（MVP）和处于学习或探索阶段的工作。其目标是追求速度和探索感，而不是关注代码的结构化、可维护性或稳定性。

Vibe Coding 与 AI-Assisted Engineering的区别

Vibe Coding：

更侧重于自由创作和快速实验，允许开发者在较少结构化的环境下快速推倒重来。

适合进行创意流动、原型设计和探索阶段的工作。

注重速度和探索，而非大规模生产环境中的可维护性和稳定性。

开发者与AI的互动较为松散，强调创意的共创。

AI-assisted Engineering：

更加接近传统的软件工程方法，要求有清晰的规划、系统化的需求说明以及可追溯的上下文。

AI在此模式中是一个强大的协作者，但始终由工程师掌控。开发者需要负责架构设计、代码审查以及系统的安全性、可扩展性、可维护性等方面。

强调专业能力的重要性，AI帮助提高开发效率，但最终的责任、判断和后果由工程师承担。

AI工具在此过程中的作用是提速和协助完成一些重复性工作，但不能替代工程师对工程原则的遵循。

注意看，这2种模式适用的人群和场景其实非常不同。Vibe Coding更是适合我这种技术小白做一些快速的想法验证。AI-assisted Engineering明显是为了辅助程序员进行系统化的工具。

Chrome 开发者体验团队的负责人Addy Osmani表示：Vibe Coding目前是只能做demo，快速验证想法用的，可以做“高保真表达”，确实非常不适合做大规模工程的产品，尤其是在一个非常有“历史”的产品中进行迭代构建，他非常反对“跟着 vibe 走”的行为，比如在遇到 Bug 时，第一反应就是：“嗨 LLM，这个按钮好像不对劲，帮我修下。”

正确的做法其实是告诉LLM，哪里不对劲，正确的方式是怎么样的，应该如何修改等，但是这种修改建议，是前端交互可能还比较好表达，对一个有一些软件开发常识、标准交互输出的产品经理可能有一定的办法，但是对开发标准语言完全不了解的普通人来说，确实是很难解决的问题。

2026年最好的Vibe Coding工具

先看结论，核心信息已浓缩至下表。沟通中，双方围绕各自实践对各大vibe coding 工具进行了多维度评价，针对技术与非技术用户区分推荐，最后总结了选择vibe coding的意见。

｜对每个平台的优缺点评价

Cursor 的开发者优先主导地位

格雷格将 Cursor 评为 A 级到 S 级之间，他强调，虽然它需要技术知识，但庞大的社区和教程生态系统使其易于使用。Cursor 团队明确是为开发者而构建，而非“氛围编码者”，但该工具的受欢迎程度已经产生了类似 React 的网络效应。卓越的智能体架构——即 Cursor 的工具如何读取、编写和编辑代码——使其即使使用与竞争对手相同的底层模型时也具有优势。

Lovable、Bolt、v0

Lovable 和 Bolt 都被评为 B 级，因为它们简化了后端复杂性，但限制了灵活性。Lovable 将用户锁定在 Supabase 中；V0 的 Vercel 市场提供了更多后端选项，包括 Convex。格雷格认为，对于那些希望原型不会在几个提示后就失效的非技术构建者来说，V0 的模板库和与 Vercel 的紧密集成使其成为更优选择。

移动氛围编码浪潮

Rourke、Vibe Code App 和 Anything 代表了一类针对移动应用开发的新平台，它们可能都基于 Expo 构建，以实现跨平台覆盖。米基认为这里有巨大的机会：解决小众问题的消费类应用（比如为高个子人群提供背痛指导）可以通过 TikTok 找到受众，并在用户有需求的时刻促成下载转化。这些平台因处于早期阶段但前景良好而获得 B 级评价。

Claude Code 近期的困境

两位主持人表示，Claude Code 最近被“削弱”了——一个月前本可以评为 S 级，现在却排在 Cursor 之后。关键见解是：使用相同的模型（Claude Sonnet 3.5）并不能保证产生相同的输出。智能体质量——即让模型能够读取、写入和编辑文件的工具层——极为重要。目前，Cursor 的智能体架构优于 Claude 自身对其模型的实现。

Codex：进步最大的模型

OpenAI 的 Codex 模型因其快速改进而获得认可，从表现平平发展到能与 Claude Sonnet 3.5 竞争。虽然在编码任务上仍略逊于 Claude，但 Codex 受益于 OpenAI 的资源以及通过 ChatGPT 的网页界面不断增长的非技术用户群。格雷格将其排在 Cursor 之后，但在 Lovable 和 Bolt 之前，并对其发展轨迹持乐观态度。

｜选择建议

技术用户建议优先选择Cursor或Cloud Code。

非技术用户可以先从Vzero、Lovable、Bolt、移动端平台等入手，但要有“做软件需要时间和反复尝试”的心态，不要期望几次prompt就能做出成品。

｜心态调整与行业观察

强调做产品/软件是一个需要耐心和反复试错的过程，不能指望AI工具一蹴而就。

  Mickey指出：“Welcome to software development.” 充满戏谑的现实提醒：出错、重试、挫败是常态。

  米基的核心建议是：非技术人员必须放弃“五个提示就能生成可投入生产的软件”这一期望。构建软件需要规划、测试、 alpha 和 beta 阶段，并将代码视为一门手艺。像 Lovable、V0 和 Bolt 这样的工具为创作提供了可能，但软件构建仍然困难——尤其是人们愿意付费的软件。首次尝试失败带来的挫败感是正常的；解决办法是在多种工具和多次迭代中坚持下去，而不是放弃。

建议关注各平台创始人的动态，选择自己信任的团队和产品。

You are betting on someone. You’re betting on a company.-->前期阶段，选择可以信任，且可以持续发展的。
