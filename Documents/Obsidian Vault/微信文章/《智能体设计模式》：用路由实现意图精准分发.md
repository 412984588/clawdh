导读：路由

你开发了一个超级智能体，它能帮你预订航班，也能解答百科知识，甚至还能提供技术支持。但如果所有用户请求都走同一条处理路径，那效率和体验可就大打折扣了。这就好比一个万能厨师，不管什么食材都用同一种烹饪方式，结果可想而知。


在构建智能体的过程中，我们常常会遇到一个核心挑战：如何让智能体在面对多样化的用户需求和环境变化时，能够灵活选择最合适的处理方式？

答案就在我们今天要导读的《智能体设计模式》第二章——“路由”。

为什么智能体需要“路由”？

在传统的程序设计中，我们习惯于线性的、一步接一步的执行流程，但这种模式显然不够用。正如书中所说：

提示链的顺序处理虽是执行确定性线性工作流的基础，但面对需要自适应响应的场景时适用性有限。现实世界的智能体系统常需基于情境因素——环境状态、用户输入或前序操作结果——在多个行动方案间做出选择。

简单来说，如果智能体没有“路由”能力，它就只能按照预设的固定步骤走下去，无法根据用户意图、环境状态等因素灵活调整。这会导致它效率低下，甚至给出不相关的回复，让用户体验直线下降。

路由：智能体的“大脑导航仪”

那么，“路由”究竟是什么呢？书中给出了精辟的定义：

路由（Routing）机制正是将控制流导向相应功能模块、工具或子流程的关键。路由为智能体引入条件判断逻辑，使其从固定流程转变为动态评估模式：根据具体情况选择最合适的后续动作，实现更灵活、更具上下文感知能力的系统行为。

你可以把“路由”想象成智能体的“大脑导航仪”。当一个用户请求进来时，这个导航仪会首先分析请求，判断用户的真实意图，然后根据分析结果，将请求精准地导向最专业的处理模块。

举个例子，书中提到的客户咨询智能体：

以客户咨询智能体为例，集成路由功能后，它先对查询分类以判断用户意图，再根据结果将查询导向相应目标：专门的问答智能体、账户信息数据库检索工具，或复杂问题升级流程，而非固守单一响应路径。

这意味着，当你说“我想查一下我的订单状态”时，路由机制会把你的请求导向“订单查询”模块；而当你说“请问如何重置密码”时，它则会导向“技术支持”流程。是不是比直接给你一堆无关信息要高效得多？

路由的实现方式

路由并非只有一种实现方式，书中介绍了多种核心机制，各有侧重：

基于大语言模型的路由（LLM-based Routing）： 最灵活的方式。智能体直接询问大模型：“这个请求应该分到哪个类别？”让大模型根据其强大的理解能力来做出判断。
基于嵌入的路由（Embedding-based Routing）： 这种方式更侧重语义理解。它会将用户请求转换为一个“数字指纹”（向量嵌入），然后与各种预设路径的“指纹”进行比对，选择最相似的路径。
基于规则的路由（Rule-based Routing）： 最直接、最确定。通过关键词匹配、正则表达式等预设规则来判断。虽然不够灵活，但在某些特定场景下（例如，明确的关键字触发特定操作）效率很高。
基于机器学习模型的路由（Machine Learning Model-Based Routing）： 这种方式通过训练一个专门的分类模型来完成路由任务，它在准确性和灵活性之间取得了很好的平衡。
动手实践：LangChain 和 Google ADK

如何将路由落地到代码中呢？

《智能体设计模式》第二章给出了详细的LangChain和Google ADK实战代码示例。

以 LangChain 为例，书中演示了一个模拟的智能体系统。它设置了一个“协调员”，根据请求意图（预订、信息查询或意图不明）将用户请求路由到不同的模拟“子智能体”。核心思路是：

意图分类： 使用一个大语言模型（如 Gemini）对用户请求进行分类，判断其属于“预订”、“信息查询”还是“意图不明”。
动态委派： 根据大模型的分类结果，通过 LangChain 的 RunnableBranch 机制，将请求委派给对应的处理函数（例如 booking_handler或 info_handler）。
# 示例代码片段（更多细节请查阅原书）
coordinator_router_prompt = ChatPromptTemplate.from_messages([
    ("system", "Analyze the user's request and determine which specialist handler should process it."),
    ("user", "{request}")
])
if llm:
    coordinator_router_chain = coordinator_router_prompt | llm | StrOutputParser()

# 定义委派逻辑
delegation_branch = RunnableBranch(
    (lambda x: x['decision'].strip() == 'booker', branches["booker"]),
    (lambda x: x['decision'].strip() == 'info', branches["info"]),
    branches["unclear"] # 默认分支
)


这段代码简洁地展示了如何利用大语言模型作为“路由器”，动态地将请求分发给不同的处理模块。Google ADK 的示例也异曲同工，通过定义不同的“工具”和“子智能体”，利用框架的自动委派机制来实现路由。

阅读指引

通过“路由”模式，我们的智能体不再是单一的执行器，而是一个能够动态思考、灵活行动的“大脑”。它能根据上下文和意图，精准地选择最合适的工具或流程，从而实现更高效、更个性化的用户交互。

正如书中所强调的：

路由赋予系统逻辑判断能力，这对构建功能多样、具备上下文感知的系统至关重要。它让智能体从按预设步骤执行的静态执行器，转变为能在环境变化时动态选择最有效方式完成任务的动态系统。

如果你希望你的智能体能够更聪明、更实用、更能应对现实世界的复杂挑战，那么掌握“路由”模式绝对是不可或缺的一步。
