这里我终于理解了他们吹的“工程化”是什么了。

敏捷开发是个啥？没事，不需要了解，下面我们来具体解读下，看完你就知道这玩意儿怎么玩了。

01. 先用语音转文字，口喷写个 PRD

这里 ralph 自带了一个写 PRD 的 skill，专门给没当过产品经理的我们用的，通过对话的方式，一步步细化细节。

推荐你用 typless 或 wisprflow 这种语音输入的工具来口喷需求，用起来巨爽，绝对回不去。想到啥就说啥，不用担心逻辑或重复，自带的那个 skill 会帮你搞定。

02. Amp 的 prd skill 会把大需求拆成好几个小需求

饭要一口一口地吃，AI 再怎么做牛做马，做需求也得一个个来。

这里核心用到的是敏捷开发里的“需求分解”的技巧，把一个看起来要做好几个礼拜的大需求，拆解成十几个互不耦合、可独立交付和验收的小需求（行内黑话叫 user story，用户故事）。

有了这些 user story，AI 就可以像勤勉的小蜜蜂一样，一个个地去把他们开发、测试、提交上线。

03. 自动化测试验收

这里你可能会好奇，AI 怎么保证写的代码没问题啊。

这里再介绍一个重要的概念，每个前面 AI 帮我们拆解完的 user story 里，都有那么一段话，描述这个需求的验收标准的。

举个例子，我在界面上放了一个输入用户名和密码的输入框，一个登录按钮。那么验收标准就得包括输入正确的信息登录成功，错误的信息则拦截报错。

有了这些信息，Amp 就可以调用自己的浏览器自动化测试工具，来模拟人类操作，输入用户名密码，点击登录。

如果某个测试没过，那就自己截图分析，看日志，直到把 bug 修复。

04. 再多需求也不怕，Ralph 可以无限循环

Amp ralph 本质是一个循环的脚本，原理也很简单。

它就像一个程序员，每次到任务看板上认领一个 user story，然后回到工位开始撸代码、写测试、测试通过，提交代码、更新状态，再去拿下一个任务。。。。

05. 怎么维护上下文记忆

在 Claude Code 的 ralph loop 插件里，它其实是用 stop hook 来阻止 Claude Code 退出，好处是可以复用之前的上下文，缺点也很明显，长时间运行时会触发自动压缩，并且大概率还有不少浪费的上下文 token。

Amp 的 ralph 则更简单纯粹，每个任务，都是独立的一个从头开始的会话，不过它自己建立了一套任务更新与记忆的机制，来确保每次都可以拿到最重要、最相关的上下文，不错过，也不浪费。

具体包括 progress.txt 和 agents.md，一个记录进度，一个类似 claude.md 是全局项目记忆，它们都会被动态更新，从而保证重要的信息像接力一样传到下个 user story 上。

大白话说，就是好记性不如烂笔头，为了省钱没那么多上下文，我写下来还不行吗。

06. 这玩意儿安全吗，会不会把我电脑硬盘都删了

昨天的文章分享后，也有人评论说，会不会有啥危险动作，可能他们看了太多删库跑路的报道了。

作为一个长期默认开着 bypass permissions 的人，我觉得这个情况下更安全了。

因为所有的需求都是你自己确认的，它再怎么改，也不会动到别的地方。
