## 完整内容

前言

前两天 Anthropic 官方推出了一个教程《Effective context engineering for AI agents》，中文叫做 AI Agent 的高效上下文工程，因为对于 AI Agent 而言，上下文（Context）是其赖以思考和行动的关键资源，但这一资源并非取之不尽的。所以这篇文章就是在深入探讨如何更高效地规划和优化 AI Agent 的上下文使用策略。下面我将以大家能看懂的大白话给大家翻译一下这篇文章，再说说我自己对于上下文功能的一些看法。

AI Agent 的制胜关键：高效上下文工程

在 AI 应用领域，“提示词工程”（Prompt Engineering）已是人尽皆知的概念。而现在，一个新的关注点出现了：我们应该如何配置上下文，才能最有效地引导模型产生我们期望的行为？

简单来说，上下文就是大语言模型（LLM）在生成内容时所能参考的全部信息。模型的“注意力”是有限的，我们必须让每一条输入的信息都物尽其用。想要高效地利用 LLM，就需要从一个整体的视角去审视它当前的“状态”，并预判这个状态将如何引导其后续行动。

本文将为你揭开“上下文工程”（Context Engineering）这门新兴艺术的神秘面纱，并提供一套实用心法，助你构建出更高效、更可控的 AI Agent。

上下文工程：提示词工程的“下一步”

在 Anthropic，我们认为上下文工程是提示词工程的自然演进。

“提示词工程”主要关注如何撰写和组织指令，以优化单次任务的输出结果。而“上下文工程”则着眼于在模型的整个推理周期中，如何动态地规划和维护一个最优的信息（令牌）组合。这个组合不仅包括最初的提示词，还涵盖了工具、对话历史、外部数据等所有动态信息。

在 LLM 应用的早期，大部分工作都围绕着提示词展开，因为当时的任务相对简单，比如文本分类或内容生成，一个精心设计的提示词就足以应对。然而，当我们开始构建能够在多轮交互中持续运行的、更强大的 Agent 时，我们就必须学会管理一个完整的“上下文状态”，包括系统指令、可用工具、外部数据、历史消息等等。

一个需要循环运行的 Agent 会不断积累大量信息，但并非所有信息都对下一步的决策有用。因此，定期“修剪”和“提炼”就变得至关重要。上下文工程的精髓，就是从一堆纷繁复杂的信息中，精准地筛选出真正有价值的部分，喂给模型的“短期记忆”。

提示词工程是一次性的任务，而上下文工程则是一个持续迭代的过程，每一次与模型交互都需要精心规划。

为什么上下文工程如此重要？

尽管 LLM 的速度和处理能力在飞速提升，但我们发现，模型和人类一样，当面对的信息过多时，也会出现“注意力分散”或“思维混乱”的情况。“大海捞针”基准测试揭示了一个名为“上下文衰减”（Context Decay）的现象：随着上下文窗口中的信息量增加，模型准确回忆起其中某个具体信息的能力会随之下降。

虽然不同模型的衰减曲线有所不同，但这个现象普遍存在于所有模型中。这告诉我们，上下文并非“多多益善”——每增加一个词，都在消耗模型有限的注意力资源。正如人类的“工作记忆”容量有限一样，LLM 在处理海量信息时也有一个“注意力预算”。你塞给它的每一个新信息，都在瓜分这个预算。

这种“注意力稀缺”源于 LLM 的底层架构——Transformer。该架构使得上下文中的每个词（Token）都能与其他所有词建立关联，这意味着 n 个词会产生 n² 级别的关系对。当上下文长度增加时，模型维持所有这些关系的能力就会变得吃力。

此外，模型是从训练数据中学习它的注意力模式的，而训练数据中短序列的文本比长序列的文本更加常见。这意味着模型天生就更擅长处理短距离的依赖关系。虽然一些技术（如位置编码插值）能让模型处理更长的序列，但这通常会以牺牲对词元位置的精准理解为代价。

“位置编码插值”是什么？

大语言模型在阅读一篇文章时，不像我们一样一个词一个词地按顺序读。它更像是一眼“看”到所有的词。那么，它怎么知道哪个词在前，哪个词在后呢？

答案就是“位置编码”。你可以把它想象成给每个词发一个隐形的“座位号”。比如，“我”“爱”“你”这三个字，模型不仅知道它们是什么字，还知道它们的座位号分别是 1、2、3。正是这些座位号，让模型理解了句子的顺序和语法。

现在，问题来了。模型在“训练”的时候，可能只学会了发 1 到 4000 号的座位。如果有一天，你给它一篇特别长的文章，有 8000 个词，那第 4001 个词之后的座位号，模型就“没见过”，它会彻底懵掉，不知道该怎么办。

“位置编码插值”就是解决这个问题的聪明技巧。

这个技巧的核心思想不是去创造新的、它没见过的 4001 号、4002 号座位。而是把原来 1 到 4000 号的座位体系，像拉伸一根橡皮筋一样，平滑地“拉长”，让它刚好能覆盖到 8000 个词的位置。

具体来说，就是对原有的座位号进行重新“校准”：

•
•原来 1 号座位可能还在原地。
•原来 2 号座位的位置，现在可能被当成是 4 号座位。
•原来 3 号座位的位置，现在可能被当成是 6 号座位。
•……以此类推。

通过这种方式，模型就可以用它“熟悉”的、有限的座位号体系，去理解一个远超它原始训练长度的文章。它不需要学习全新的东西，只需要把已有的认知“等比例放大”或“稀释”一下即可。

种种因素叠加，导致了模型性能一种“平滑下降”而非“断崖式下跌”的趋势：在长上下文中，模型依然强大，但在信息检索和长距离推理的准确性上，却不如短上下文来得可靠。

正是这些现实，使得精心的上下文工程对于构建强大的 Agent 来说，变得至关重要。

高效上下文的几大要素

既然 LLM 的注意力预算有限，那么优秀的上下文工程，就是要找到一个“信息量最大、但体积最小”的令牌集合，用最高效的方式去激发模型达成我们期望的目标。知易行难，下面我们来看看如何将这个原则应用到上下文的各个组成部分中。

系统提示词（System Prompts）

系统提示词应当极其清晰，语言直白，并且在恰当的抽象层级上表达意图。所谓“恰当的层级”，是为了避免两种常见的陷阱：

1.过于具体：一些工程师喜欢在提示词里硬编码复杂脆弱的 if-else 式逻辑，试图精确控制 Agent 的每一步行为。这种方式不仅维护成本高，而且极易因微小变化而失效。
2.过于模糊：另一些工程师则只提供高度概括的指导，没有给出期望输出的具体信号，或者错误地假设模型与自己拥有共同的背景知识。

最佳的抽象层级应该介于两者之间：既要足够具体，能有效引导模型的行为；又要足够灵活，为模型留下一定的自主发挥空间。

理想的提示词应避免两个极端：一端是脆弱的硬编码逻辑，另一端是过于笼统或想当然的指令。

我们建议将提示词结构化，比如使用 XML 标签（如 <背景信息>、<指令>）或 Markdown 标题来划分不同部分。当然，随着模型越来越智能，提示词的具体格式可能不再那么重要，但清晰的结构总是有益的。

无论你如何设计，目标都应该是用最精炼的信息，完整地描述出期望的行为。（请注意，精炼不等于简短，你需要提供足够的信息来确保 Agent 的行为符合预期。）最佳实践是：从一个最简单的提示词开始测试，观察模型的表现，然后针对其不足之处，逐步补充清晰的指令和示例。

工具（Tools）

工具是 Agent 与外部世界交互的桥梁。设计精良的工具不仅能为 Agent 带来关键信息，还能鼓励其形成高效的行为模式。

我们发现，最常见的问题之一是工具集过于臃肿，功能重叠，让模型难以抉择。试想，如果连人类工程师都无法判断在某个场景下该用哪个工具，我们又怎能期望 AI Agent 做得更好呢？一个设计良好的工具集，应该像优秀代码库里的函数一样：功能独立、边界清晰、错误处理完善。

后续我们还会讨论到，维护一个“最小可用工具集”，对于长期交互中的上下文维护和修剪也大有裨益。

示例（Few-shot Prompting）

提供示例，也就是“少样本提示”，是一种广为人知的最佳实践。但我们发现，很多团队倾向于在提示词里塞满各种边界情况的示例，试图覆盖所有可能遇到的规则。我们不推荐这种做法。

相反，你应该精心挑选一小组具有代表性的、多样化的示例，让模型能从中“悟”出你期望的行为模式。对 LLM 而言，好的示例胜过千言万语。

总而言之，我们对上下文所有组成部分的指导原则是：深思熟虑，保持信息丰富度的同时，尽可能紧凑。接下来，让我们深入探讨如何在运行时动态地管理上下文。

上下文检索与“Agent 式搜索”

我们曾将 Agent 定义为：在循环中自主使用工具的 LLM。随着底层模型日益强大，Agent 的自主性也在不断扩展，更智能的模型能够独立地处理复杂问题并从错误中恢复。

这也带来了工程师在设计上下文策略上的思维转变。如今，许多 AI 应用都采用基于“向量嵌入”的检索方式，在推理前就把相关资料提取出来，喂给 Agent。而随着 Agent 模式的普及，我们看到越来越多的团队开始采用一种更动态的“即时”（Just-in-Time）上下文策略来增强系统。

在这种模式下，Agent 不再需要预先加载所有可能用到的数据。相反，它只维护一些简单的标识符（如文件路径、数据库查询语句、网页链接等），并在需要时通过工具动态地将数据加载到上下文中。这就好比人类工作时，不会把整座图书馆都记在脑子里，而是通过文件系统、搜索引擎、书签等方式，在需要时按需查找信息。

这种方法的元数据本身就提供了宝贵的行为线索。比如，一个位于 tests/ 目录下的 test_utils.py 文件和一个位于 src/ 目录下的 core_logic.py 文件，其用途不言而喻。文件夹结构、命名规范、时间戳等，都在为人类和 Agent 提供着如何以及何时使用这些信息的重要信号。

让 Agent 自主地探索和检索数据，还解锁了“渐进式探索”的能力。Agent 可以通过与环境的交互，一步步构建起对问题的理解。每一次交互都产生新的上下文，为下一步决策提供信息：文件大小暗示了其复杂性，命名约定揭示了其用途，时间戳可能代表了其相关性。通过这种方式，Agent 可以层层递进地构建认知，只在“工作记忆”中保留最必要的信息，并利用笔记等策略进行信息持久化。

当然，这种方式也有取舍：运行时的动态探索比预先检索计算好的数据要慢。而且，它需要开发者精心设计工具和启发式规则，以确保 Agent 能够高效地在信息环境中导航，否则 Agent 可能会误用工具、钻牛角尖，浪费宝贵的上下文。

在某些场景下，混合策略可能是最高效的。Agent 可以预先检索一部分核心数据以保证速度，同时保留自主探索的能力。具体采用何种策略，取决于任务的性质。

应对“长任务”的上下文工程挑战

对于需要长时间运行的任务（例如大型代码库迁移、全面的课题研究等），Agent 必须在远超上下文窗口长度的动作序列中保持连贯。为了解决这个难题，我们开发了几种核心技术：压缩、结构化笔记和子 Agent 架构。

1. 压缩（Compression）

当对话历史即将撑满上下文窗口时，我们可以让模型对其进行总结，然后用这个浓缩后的摘要开启一个新的、干净的上下文窗口。压缩的关键在于精准提炼，确保 Agent 在丢失最少信息的情况下继续工作。

例如，在我们的编码助手 Claude Code 中，当历史消息过多时，模型会自动总结其中最关键的细节，比如架构决策、未解决的 bug、实现要点等，同时丢弃冗余的工具输出或过程性对话。然后，Agent 会带着这个压缩后的上下文，加上最近访问过的几个文件，继续它的工作。

实现压缩时，最大的挑战在于决定保留什么、丢弃什么。过于激进的压缩可能会丢失一些当时看似无用、但后续至关重要的细节。一个简单有效的起点是清理掉旧的工具调用和返回结果。一旦工具执行完毕，Agent 往往不再需要看到原始的返回数据。

2. 结构化笔记（Structured Note-Taking）

这是一种让 Agent 定期将关键信息和思考过程记录到外部存储（例如一个 NOTES.md 文件）的技术。当需要时，再将这些笔记重新读入上下文。

这种策略以极小的开销实现了记忆的持久化。就像 Claude Code 会创建待办事项列表一样，这个简单的模式能让 Agent 在复杂任务中追踪进度，记住那些在数十次工具调用后早已被冲出上下文窗口的关键依赖和信息。

在“Claude 玩宝可梦”这个有趣的实验中，Agent 通过笔记在数千个游戏步骤中保持了惊人的连贯性。它会记录“过去1234步，我一直在1号公路练级，皮卡丘离10级目标还差2级”这样的信息。它自发地绘制探索过的地图，记录关键成就，并总结战斗策略，从而不断学习如何克制不同的对手。每当上下文重置后，它只需读一遍自己的笔记，就能无缝衔接之前的任务。

3. 子 Agent 架构（Sub-Agent Architecture）

这是一种“分而治之”的策略。与其让一个 Agent 独自承担所有任务，不如将其分解，让一个“主 Agent”负责高级规划和协调，并将具体的执行任务分配给多个“子 Agent”。

每个子 Agent 都在一个干净的上下文窗口中专注于自己的任务，比如深度搜索、代码实现等。它们可能会在自己的任务中消耗大量令牌，但最终只需向主 Agent 返回一个高度浓缩和提炼的摘要（比如几千个令牌）。这种模式通过明确的责任划分，将复杂的探索过程隔离在子 Agent 内部，而主 Agent 则能专注于整合与分析最终结果，极大地提升了处理复杂研究任务的效率。

这几种方法如何选择，取决于任务的特点：

•压缩：适用于需要大量来回对话、保持流畅性的任务。
•笔记：在有明确里程碑的迭代式开发或探索任务中表现出色。
•子 Agent：非常适合那些可以通过并行探索获益的复杂研究与分析任务。

结语

上下文工程代表了我们构建 LLM 应用方式的一次根本性转变。随着模型能力越来越强，我们的挑战不再仅仅是设计一个完美的提示词，而是要在 Agent 的每一步行动中，都深思熟虑地为其有限的“注意力预算”规划最佳的信息组合。

无论你是在为长任务设计压缩策略，还是在为 Agent 打造高效的工具集，亦或是让它在环境中即时探索，背后的指导原则始终如一：找到能够最大化达成期望目标的、那个信息量最大但体积最小的令牌集合。

我们相信，随着模型持续进化，这些技术也将不断演进。更智能的模型将需要更少的人工干预，从而拥有更大的自主性。但即便如此，将上下文视为一种宝贵而有限的资源，仍将是构建可靠、高效 AI Agent 的核心准则。

AI 的“注意力”是我们最需要管理的产品资源

文章最关键的洞察，是把 LLM 的上下文窗口类比为人类的“工作记忆”或“注意力预算”。这个比喻一下子点醒了我。

作为产品经理，我们天天都在跟“稀缺资源”打交道：屏幕空间、用户的时间、开发资源等等。现在，这篇文章告诉我们，AI 的“注意力”也是一种有限且昂贵的资源。

这个视角彻底改变了我的产品思路：

•过去：我们总觉得给 AI 的信息越多越好，生怕它“不知道”，于是拼命地把文档、历史记录、API 说明全都塞进 Prompt 里。这就像一个焦虑的管理者，把所有文件都扔给下属，然后期望他能自己理清头绪。
•现在：我意识到，我需要像设计用户界面一样，去设计 AI 的信息流。我的工作不再是“信息投喂”，而是“信息策展”。我需要思考，在任务的每一步，到底哪几条信息是“高信噪比”的？哪些是能四两拨千斤的关键指令？

对于产品经理来说，最怕的就是问题提出来了，却没有解决方案。这篇文章的价值在于，它不仅指出了问题，还给了我们一个非常实用的“工具箱”，用来解决 AI Agent “又笨又贵”的难题。

•应对“笨”（提升用户体验）：用户最常抱怨 AI “金鱼记忆”，聊着聊着就忘了前面的事。文章里的“压缩”和“结构化笔记”技术，就是解决这个问题的方法。在产品设计上，我就可以规划：当对话超过一定长度时，自动触发一个“记忆巩固”机制；或者设计一个“备忘录”功能，让 Agent 可以随时记录和查阅关键信息。这能极大提升产品的连贯性和智能感，让用户觉得这个 Agent “靠谱”。
•应对“贵”（控制运营成本）：API 调用是按 token 计费的，超长上下文的成本非常惊人。文章提出的“即时检索”（Just-in-Time）和“子 Agent 架构”，就是控制成本的利器。“即时检索” 意味着我不需要在一开始就把一个 100 页的 PDF 全部喂给 AI，而是让 AI 先学会“查目录”，需要哪部分再去看哪部分。这在产品层面，就是从“一次性加载”模式转向了“按需加载”模式，成本能指数级下降。“子 Agent 架构” 更是给了我设计复杂产品的蓝图。我可以把一个庞大的任务（比如“帮我规划一次欧洲旅行”）拆解成多个子任务，交给不同的“专家 Agent”（一个负责搜机票，一个负责订酒店，一个负责规划路线），最后由一个“项目经理 Agent”来汇总。这样做不仅每个 Agent 的上下文都更小、更专注、成本更低，而且整个系统的健壮性和可维护性也大大提高了。

未来的竞争，可能不再是比谁家的模型“参数更大”或“上下文窗口更长”，而是比谁的“上下文工程”做得更出色。因为单纯扩大窗口，就像给一个注意力不集中的人一本更厚的书，问题并没有解决。

高效的 Agent 应该像一个经验丰富的人类专家：

•知道什么时候该“专注”：只把最关键的信息放在手边。
•知道什么时候该“查询”：懂得利用外部工具和知识库。
•知道什么时候该“求助”：懂得把任务分解给更专业的“同事”（子 Agent）。
•知道什么时候该“复盘”：会自己做笔记，总结经验。
