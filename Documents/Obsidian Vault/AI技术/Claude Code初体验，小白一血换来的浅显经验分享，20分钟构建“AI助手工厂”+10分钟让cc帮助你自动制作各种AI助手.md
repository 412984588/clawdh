第一：即使是已经配置了性价比较高的kimi k2模型，也还是要注意token消耗。因为agent编程真的真的真的比单纯的chat对话耗费的tokens多的多的多得多！！！我刚开始想着试试无脑使用claude code来测试一下他的能力，就直接无脑开始对CC提出需求让它自己规划自己完成自己测试自己修改，进行了十几轮对话之后勉强完成基本功能，我觉得效果不满意，就又开了新的项目开始新的需求，十几轮对话之后也是勉强能有，还在改bug呢然后竟然就开始报错tokens不足了，一下子干掉我65块钱的tokens啊，心疼。这还是使用性价比较高的模型。唉，感觉也没有比充值claude会员便宜多少。然后没办法继续找能够降低成本的方法，幸好claude 的社区很活跃，找到了一个github项目Claude Code Router可以给claude code添加代理路由，根据需求自由切换各种大模型。超级实用，下一篇文章我将分享详细的配置方法。
第二：基于以上token消耗花费巨大的原因，我建议大家在使用claude code之前一定要先大致学习一下官方文档，以提高工作效率，避免不必要的tokens消耗。学习请戳：
小白的AI自动化专家转型计划ClaudeCode+SubAgent+MCP+n8n+Context/PromptEngineer
或者直接去官网开始：https://docs.anthropic.com/en/docs/claude-code/overview
不必找其他的教程，一定先学习官方文档。在官方文档中你会学到claude code中CLAUDE.md上下文文件，subagent子代理设置，outputstyle文件设置，hooks钩子设置，MCP配置的基本用法，然后就会形成自己的构建claude code助手的大致脉络了。我使用claude code有两个大致的场景，第一让claude code作为你的编程程序员助手，给你编写其他项目的代码，构建网站网页、app、AI agents、游戏等项目代码。这个因为我的开发经验不够丰富也没有需求，我没有在这个场景上进行深入测试。第二：通过设置claude code的上下文prompts以及设置对应的subagents群，来打造自己的以claude code为CPU的专业助手，通过建立多个项目目录以及开启多个claude code对话来运行多个助手相互协作，我自己就是在这个场景下进行了多个需求测试。
搭建助手的第零步就是首先搭建了一个帮助我规划设计 构建claude code助手的 助手。他的主要职能就是当我提出构建某种助手的需求时帮助我分析需求并直接构建claude code 助手。这里需要说的是，因为我真的很懒，所以我并没有进行过多的分析和思考。我还是直接打开claude code向他描述了我的需求，请他自己构建自己。构建助手的助手过程如下：这个助手我称之为“助手工厂”
第三：构建claude code助手要注意几点准则：第一清晰的描述你的功能需求，这个清晰指你需要明白自己想要的助手具有什么功能，最终需要给出什么结果。如果你不能清晰的描述，可以先和deepseek，chatgpt，豆包等其他大模型进行讨论对话，让大模型帮助你厘清需求，创建高效的prompts。
然后根据官方文档的claude  code使用指南，构建时一定要先开始规划再开始编码，在规划时要明确指令“现在不要编码”。直到他给出你满意的规划之后再开始设置编码指令。否则会浪费tokens。
在claudecode完成之后，要自己在新项目中使用真实的需求进行测试，不能完全依赖claude code自动测试。我有两次遇到自动测试通过，但直接上手使用就糟糕的。自己测试之后一定要详细记录表现出的问题，比如没有按照要求输出、缺少功能、输出质量低下、配置文件目录不对等等。将这些问题和你想如何改进详细反馈给之前的claude code构建助手，让他帮你进行迭代，多迭代几次基本能达到你的要求。注意一个小细节，这里测试的时候是在你的测试项目中测试的，迭代的时候是在“助手工厂”中迭代，也就是此时你需要开启两个不同的claude code对话窗口。在“助手工厂”对你的新助手进行迭代之后，要退出重启新助手的claude code，然后继续测试迭代，如此循环直到达到满意的效果。
